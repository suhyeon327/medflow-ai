# ADR-002: Soft Delete Strategy

## Status

Accepted

## Date

2026-07-02

---

## Context

의료 시스템에서는 예약, 진료 기록, 문진 결과 등의 데이터가 서로 긴밀하게 연결되어 있다.

사용자가 서비스를 탈퇴하거나 관리자가 데이터를 삭제하더라도 기존 예약 및 진료 기록은 감사(Audit)와 이력 관리 목적으로 보존되어야 한다.

물리 삭제(Hard Delete)를 사용할 경우 다음과 같은 문제가 발생할 수 있다.

- 예약 기록 손실
- 진료 기록 손실
- FK 참조 무결성 문제
- 감사 로그 확인 불가

---

## Decision

모든 주요 Entity는 Soft Delete를 적용한다.

Entity에는 다음 컬럼을 추가한다.

```
deleted_at
```

삭제 시 실제 DELETE 문을 수행하지 않고 삭제 시각을 기록한다.

조회 시에는 삭제되지 않은 데이터만 조회한다.

---

## Consequences

### 장점

- 데이터 이력을 보존할 수 있다.
- 의료 데이터의 감사(Audit)가 가능하다.
- 예약 및 진료 기록의 참조 무결성을 유지할 수 있다.
- 삭제된 데이터의 복구가 가능하다.

### 단점

- 모든 조회에서 삭제 여부를 고려해야 한다.
- 인덱스 설계가 복잡해질 수 있다.
- 데이터가 계속 누적된다.

---

## Alternatives Considered

### Hard Delete

```
DELETE FROM patient
```

장점

- 구현이 단순하다.
- 저장 공간을 절약할 수 있다.

단점

- 데이터 복구 불가능
- 의료 데이터 관리에 부적합
- 감사 로그 관리 어려움

---

## Decision Rationale

의료 서비스는 데이터의 정확성과 추적 가능성이 매우 중요하다.

사용자 삭제 이후에도 예약 기록, 진료 기록, 문진 기록은 보존되어야 하므로 Soft Delete를 채택하였다.

향후 감사 로그(Audit Log) 기능과도 자연스럽게 연계할 예정이다.
